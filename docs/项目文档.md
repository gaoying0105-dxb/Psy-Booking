## 1. 整体架构：一张图看懂

```
 浏览器（用户/管理员）
        │
        ▼
 ┌─────────────────────┐
 │  Vercel              │   托管 Next.js 前端 + 中间件
 │  Next.js App Router  │   （页面渲染、登录态保护）
 └─────────┬───────────┘
           │  supabase-js（HTTPS 直连）
           ▼
 ┌─────────────────────┐
 │  Supabase            │
 │  ├─ Auth     登录注册 │
 │  ├─ Postgres 数据库   │ ← RLS 行级安全 + 业务函数
 │  └─ 自动生成的 REST API│
 └─────────────────────┘
```

**关键认知：这个项目没有传统意义上的"后端服务器"。**
前端用 `supabase-js` 直接连数据库。那安全怎么办？靠两样东西：

1. **RLS（Row Level Security，行级安全）**：数据库给每张表定规则，比如"用户只能 select 自己的预约"。哪怕有人打开浏览器控制台直接调 API，也只能拿到自己的数据。
2. **数据库函数（RPC）**：预约、取消这种有复杂规则的操作，不允许前端直接 insert/update，而是调用数据库里的函数（`book_appointment`、`cancel_appointment`），所有校验在函数里强制执行，**前端绕不过去**。

这是 Supabase 应用区别于传统"前端 → 自己写的后端 API → 数据库"的核心思想。

## 3. 数据库设计（5 张表）

```
auth.users (Supabase 内置)
    │ 1:1（注册时触发器自动创建）
    ▼
profiles ─────────┐ 用户资料 + role(user/admin)
                  │ 1:N
                  ▼
appointments ◄─── time_slots     预约记录 N:1 时段
site_settings                    键值配置（预约须知）
operation_logs                   操作日志
```

### 几个值得学习的设计点

**① 用部分唯一索引防止重复预约（并发安全）**

```sql
create unique index one_active_appointment_per_slot
  on appointments (slot_id)
  where status in ('pending', 'confirmed');
```

含义：同一时段最多只能存在一条"进行中"的预约。就算两个人在同一毫秒点提交，数据库也只会让一个成功——这比在代码里"先查再插"可靠得多（先查再插存在时间差，并发时会双双通过检查）。已取消/已拒绝的记录不占用索引，所以取消后时段自动恢复可约。

**② 状态机思维**

预约状态只有 5 种：`pending → confirmed / rejected / cancelled`，`confirmed → no_show / cancelled`。"时段是否可约"不单独存字段，而是实时推导：*开放的时段 && 没有进行中的预约 = 可约*。少存一份状态，就少一种数据不一致的可能。

**③ 时区统一**

所有"距开始还有多久"的判断都先把 `日期+时间` 解释为北京时间（`Asia/Shanghai`）再比较。前端 `format.ts` 里同样用 `+08:00` 后缀构造时间。这样无论服务器在哪、访问者在哪个时区，48 小时规则的判定都一致。

## 4. 核心业务规则在哪实现

| 需求文档里的规则 | 实现位置 | 怎么实现 |
|---|---|---|
| 时段冲突检查 | schema.sql 唯一索引 + `book_appointment` | 索引兜底，函数里先 `for update` 锁行再检查 |
| 必填项/手机号/邮箱校验 | 前端 `book/page.tsx` + 函数内正则 | 前端校验为了体验，数据库校验为了安全（双重）|
| 48 小时取消限制 | `cancel_appointment` 函数 | `slot开始时间 - now() > 48h` 否则抛中文异常 |
| 不足48h由咨询师取消 | 管理后台 + RLS | 管理员 update 不走该函数，无时间限制 |
| 失约 2 次禁约 | `book_appointment` 函数 | 预约前 count 该用户 no_show 记录 |
| 屏蔽/解除屏蔽时段 | `time_slots.status` 字段 | blocked 的时段不出现在 `get_available_slots` 结果里 |
| 预约须知可编辑 | `site_settings` 表 | 后台改，确认页读 |
| 区分普通用户/管理员 | `profiles.role` + `is_admin()` | RLS 策略、页面守卫、导航栏都基于它 |
| 操作日志 | `operation_logs` 表 | 预约/取消时函数内写入 |

**一个安全细节**：`profiles` 表用户可以更新自己的资料，但有个触发器 `prevent_role_escalation` 禁止非管理员修改 `role` 字段——否则任何人都能把自己改成 admin。写 RLS 时要始终想："如果有人直接拿 anon key 调 API，最坏能干什么？"

## 5. Next.js App Router 关键概念（看代码前必读）

**服务端组件 vs 客户端组件**
- 默认所有组件在服务端渲染，可以直接读 cookie、连数据库，如 `admin/layout.tsx`（它在服务端校验管理员身份，没权限的请求根本到不了页面）。
- 文件顶部写 `"use client"` 的是客户端组件，能用 `useState`、点击事件，如所有表单页面。

**为什么有两个 Supabase 客户端（client.ts / server.ts）？**
浏览器和服务器读写 cookie 的方式不同。登录态存在 cookie 里，`@supabase/ssr` 这个包负责让两端都能正确读到"当前是谁登录了"。规则很简单：`"use client"` 组件里用 `client.ts`，服务端组件/Route Handler 里用 `server.ts`。

**middleware.ts 干了两件事**
1. 每次请求时刷新 Supabase 会话（token 过期自动续）；
2. 拦截未登录用户访问 `/book` `/me` `/admin`，重定向到登录页并带上 `?next=` 参数，登录后跳回原页面。

**数据流转示例：用户点"提交预约"之后**

```
book/page.tsx 前端校验（手机号格式等）
  → supabase.rpc("book_appointment", {...})        # HTTPS 调用
    → Postgres 函数执行：登录校验 → 失约黑名单 →
      锁行 → 时段有效性 → 冲突检查 → insert
    → 任一步失败：raise exception '中文错误信息'
  → 前端拿到 error.message 直接展示给用户
  → 成功：拿到新预约 id，跳转 /book/success?id=xx
```

## 6. 免费额度与保活

| 资源 | 免费额度 | 本项目预估 |
|---|---|---|
| Supabase 数据库 | 0.5 GB | 一条预约约 0.5KB，能存几十万条 |
| Supabase MAU | 50,000 | 校园场景绰绰有余 |
| Vercel 带宽 | 100 GB/月 | 纯文字站点，足够 |

唯一要处理的是 **Supabase 免费版 7 天无请求会暂停数据库**。方案：`.github/workflows/keep-alive.yml` 用 GitHub Actions 每周一、四定时发一个查询请求（查 `site_settings` 表一行数据），成本为零。配置见部署指南第 5 步。

## 7. 从这里继续学习/迭代的路线

**改文案/样式（最简单）**
- 首页介绍：`src/app/page.tsx`
- 配色：`src/app/globals.css` 顶部的 CSS 变量（`--pine` 是主色）
- 时段模板：`src/app/admin/slots/page.tsx` 的 `TEMPLATE`

**加功能（按难度排序）**
1. 预约状态变更邮件通知：Supabase Database Webhooks → Resend API
2. 管理后台导出预约记录为 CSV（前端就能做：查出来拼字符串下载）
3. 多咨询师支持：加 `counselors` 表，`time_slots` 加 `counselor_id` 外键——数据结构已为此留好扩展空间（这正是需求文档"系统扩展能力"一节的要求）
4. 评价反馈：预约完成后加 `reviews` 表

**值得深入的概念（每个都查 15 分钟资料）**
- Postgres RLS：Supabase 安全的根基
- `security definer` vs `security invoker`：为什么查可约时段的函数要用 definer
- 数据库事务与行锁（`for update`）：并发预约为什么不会出错
- Next.js 服务端/客户端组件的边界

## 8. 技术选型为什么是这样

| 选择 | 理由 | 放弃的备选 |
|---|---|---|
| Next.js | Vercel 原生支持、生态最大、一套代码前后端 | Vite+React（需另找后端）|
| Supabase | Auth+DB+RLS 一站式，免费额度慷慨 | Neon（无 Auth/Storage，要拼服务）、Firebase（NoSQL，关系型数据不顺手）|
| 数据库函数承载业务逻辑 | 规则不可绕过、并发安全、省一层 API | Next.js API Routes（也可以，但多一层且仍要防直连）|
| 邮箱密码登录 | 零配置零成本 | 微信登录（Web 端需企业认证 + 300 元/年）、短信（要花钱）|
| 不用 UI 组件库 | 项目小，Tailwind 够用且便于学习 | shadcn/ui、Ant Design |
